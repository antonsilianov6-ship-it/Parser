# Правила разработки проекта

## 1. Архитектура и масштабируемость

- Всегда писать только удобный и масштабируемый код
- Избегать дублирования строк кода
- Каждый модуль должен содержать максимум 350 строк
- При превышении лимита - разбивать модуль на несколько файлов
- Использовать принципы DRY (Don't Repeat Yourself)
- Выносить повторяющуюся логику в отдельные функции/модули

## 2. Язык общения

- Все ответы в чате ТОЛЬКО на русском языке
- Не переходить на английский язык в общении
- Комментарии в коде могут быть на английском (стандартная практика)
- Документация и объяснения - строго на русском

## 3. Тестирование и исправление ошибок

- Все тесты должны исправляться сразу при их обнаружении
- Не откладывать исправление ошибок на потом
- Если ошибки появились на любом этапе Spec - исправлять немедленно
- Если ошибки пропущены на предыдущем этапе - исправлять на текущем этапе сразу же
- Не накапливать технический долг
- Каждый этап должен завершаться с рабочими тестами
- Не оставлять исправление ошибок на итоговый этап

## 4. Документация

- Не создавать лишнюю документацию после выполнения задачи
- Создавать документацию только по явному запросу пользователя
- Избегать создания README, SUMMARY и других документов без необходимости
- Фокусироваться на коде, а не на документировании процесса

## 5. Завершение задач

- Если реализуем что-то не в рамках SPEC - доводить до конца, не оставлять как TODO
- Не откладывать реализацию функционала на будущее
- Каждая начатая задача должна быть полностью завершена
- Избегать частичных решений и заглушек
- Если начали рефакторинг - завершить его полностью

## 6. Выбор решений

- Всегда выбирать самый правильный и оптимальный путь, а не самый простой
- Думать о долгосрочной перспективе и масштабируемости
- Избегать "быстрых" решений, которые приведут к проблемам в будущем
- Оптимизировать код с учетом производительности и поддерживаемости
- Предпочитать архитектурно правильные решения, даже если они требуют больше времени
- Учитывать возможные последствия и edge cases при выборе подхода
- Не жертвовать качеством ради скорости реализации

## 7. Виртуальное окружение

- ВСЕГДА использовать виртуальное окружение (venv) для запуска тестов и выполнения кода
- Перед запуском любых Python команд активировать виртуальное окружение
- Для Windows использовать: `venv/Scripts/activate` или `./venv/Scripts/activate`
- Для Linux/Mac использовать: `source venv/bin/activate`
- Все команды pip, pytest, python должны выполняться внутри активированного venv
- Не запускать тесты и код в глобальном окружении Python
- При установке зависимостей всегда использовать venv: `./venv/Scripts/pip install ...`
